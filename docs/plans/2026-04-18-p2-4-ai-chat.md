## Why

Third path to a recipe: conversational. User describes ingredients / constraints ("Kartoffeln, Quark, Lauch — 30 Min, vegan"), AI suggests. Iterate. Click "In Rezept umwandeln" → structured JSON → Review screen.

## Scope

### 1. Endpoints

`POST /chat` — turn in a conversation:
```json
{
  "session_id": "uuid",
  "messages": [
    { "role": "user", "content": "Ich hab Kartoffeln, Quark, Lauch" },
    { "role": "assistant", "content": "..." },
    { "role": "user", "content": "Vegan bitte" }
  ]
}
```
Response:
```json
{ "assistant_message": "..." }
```

`POST /chat/{session_id}/to-recipe` — convert current chat history to structured recipe:
Request body: same `messages` array.
Response: same `ExtractionResult` shape as P2-2 / P2-3.

### 2. Session persistence

**Decision (documented deviation possibility):** start with **in-memory session store with TTL** for simplicity. The client sends the full `messages` array on every turn (small — max ~20 turns × 300 chars), so the service can be stateless for the MVP. If hoppr's pattern proves otherwise when we look at it, we can add Redis-backed persistence later.

- No DB table, no Redis key.
- Session state = the `messages` array the client resends. Python service just relays.
- `session_id` is for client-side grouping (e.g. in the URL) and for logging correlation.

### 6. Tests

Always-on (mock provider):
- `POST /chat` with 1-message dialogue → assistant reply returned.
- `POST /chat` with 5-turn dialogue → last turn is user's, provider called with all 5 messages.
- `POST /chat/{sid}/to-recipe` with a dialogue that settled on a recipe → structured JSON returned.
- Empty `messages` → 400.
- `messages` with 30 turns → 413 "zu lang" (safety cap).
- Provider raises `LLMProviderError` → pipeline surfaces 503 with German message.
