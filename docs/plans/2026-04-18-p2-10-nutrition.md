# P2-10 — Nutrition Estimation

**Slice:** P2-10 (final Phase 2 slice)
**Status:** planned
**Date:** 2026-04-18
**Depends on:** P2-2 (extends structuring prompt + schema).
**Parent plan:** `docs/plans/2026-04-18-phase-2-architecture.md`.

## Why

Per PRD §5.4: LLM estimates nutrition per portion at save time. Displayed with "geschätzt" badge; user can manually override. No BLS / OpenFoodFacts dependency.

## Scope

### 1. Schema extension

`src/extractor/prompts/recipe_extraction.py` `RECIPE_SCHEMA`:
- Add optional `nutrition_estimate: { kcal: int, protein_g: int, carbs_g: int, fat_g: int }` per portion.
- Update `SYSTEM_PROMPT_DE` to request nutrition estimation explicitly.

Same for the photo schema in `src/extractor/prompts/photo_recipe.py`.

### 2. Python pipeline

`ExtractionResult.recipe` gains optional `nutrition_estimate` field. Post-processor clamps to sane ranges (kcal 0..5000, macros 0..500g) as defence against hallucinations.

Tests:
- Schema valid with nutrition field.
- Post-process preserves nutrition unchanged when in range.
- Out-of-range values get clamped.

### 3. .NET entity + migration

- `Recipe` entity gains `NutritionEstimate` JSONB column (nullable). Shape matches the Python result.
- EF Core migration: `AddRecipeNutritionEstimate`.
- On recipe-save via the import flow (P2-7/P2-8/P2-9), `nutrition_estimate` from `ExtractionResult` is persisted.
- Manual edit endpoint: `PATCH /api/recipes/{id}/nutrition` (admin OR author) with the four fields.

### 4. Web UI

- `RecipeDetailPage` shows a small "Nährwerte (geschätzt)" section below the ingredient list:
  - kcal / Eiweiß / Kohlenhydrate / Fett per Portion.
  - Each value has an edit affordance (click to inline-edit) for the author/admin.
  - "Geschätzt"-badge next to the heading.
- `RecipeFormPage` shows the nutrition block when prefilled from an import (read-only preview; user can edit before save).
- Empty/null nutrition → section hidden entirely.

### 5. Tests

Web:
- `RecipeDetailPage.test.tsx`: nutrition shown when present, hidden when null, edit flow works.
- `RecipeFormPage.test.tsx`: prefill includes nutrition, save persists it.

.NET:
- Migration applies cleanly.
- `PATCH /api/recipes/{id}/nutrition` happy + 403 + 404 + validation tests.

Python:
- Schema + post-process tests above.

## Non-goals

- No per-ingredient breakdown (just per-portion totals).
- No BLS / OpenFoodFacts integration.
- No historical trend view.
- No alerts / dietary flags.

## Acceptance criteria

- Python, .NET, web gates all green.
- Migration applies to prod-shaped DB without data loss.
- Manual: import a recipe from URL → detail page shows estimated nutrition → edit one value → save → persists on refresh.

## Anti-shortcut reminders

- TDD all three layers.
- Nutrition is optional — don't require it in any existing recipe-save path.
- Migration must be forward-only (no down-migration needed per project conventions; check existing pattern).
- Clamp LLM hallucinations: kcal > 5000 per portion likely wrong, clamp to the realistic ceiling.
- `nutrition_estimate` column nullable. Existing recipes have null.
